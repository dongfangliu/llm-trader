> 目标：把当前版本收敛为“可安全商用、可持续运维、可审计回溯”的正式上线版本。  
> 优先级定义：**P0=上线阻断项（必须完成）**，**P1=强烈建议上线前完成**，**P2=上线后迭代**。

## 2. 上线阻断项（P0，必须全部完成）

## P0-1 Ko-fi 支付链路生产化（必须）
- [ ] **Webhook 验签**：校验 Ko-fi 官方 `verification_token`（服务端环境变量），未通过直接拒绝。
- [ ] **事件幂等**：按 `kofi_transaction_id / kofi_reference_id` 建唯一键，重复回调不重复加权益。
- [ ] **事件类型白名单**：仅处理订阅创建、续费成功、取消、退款等已定义事件。
- [ ] **套餐映射硬编码表**：Ko-fi tier/金额/币种 -> `basic/premium`，不允许模糊匹配。
- [ ] **订阅状态机**：`active / past_due / canceled / refunded / expired`，并定义降级策略。
- [ ] **失败补偿**：Webhook 处理失败入库（dead-letter），支持重放补偿。
- [ ] **审计日志**：记录原始事件ID、映射结果、设备ID、生效时间、操作者（系统/人工）。
- [ ] **手动对账工具**：按日导出“Ko-fi事件 vs 本地订阅状态”差异。

**验收标准**
- 回放同一Webhook 10次，订阅状态只变更一次；
- 随机伪造 token 的请求全部 401/403；
- 续费、取消、退款三类事件能正确影响剩余次数与订阅状态；
- 任一失败事件可在后台重放成功并留痕。

## P0-2 微信登录正式接入（必须）
- [ ] 接入微信 OAuth（`code -> access_token/openid/unionid`）真实流程。
- [ ] 前端回调页处理 `code/state`，后端校验 `state/nonce` 防 CSRF。
- [ ] 统一账号标识策略（openid/unionid 与 device_id 的绑定规则）。
- [ ] 游客模式与微信模式边界清晰（升级后权益不丢失）。
- [ ] 登录失败/授权取消/接口超时 的可观测错误处理。

**验收标准**
- 真机微信授权登录成功率 > 99%（样本测试）；
- 同一微信账号跨设备登录，订阅权益一致；
- 授权取消、过期 code、重复 code 全部有明确错误反馈。

## P0-3 安全与风控（必须）
- [ ] 后端关键环境变量改为生产密钥管理（JWT secret、Ko-fi token、LLM keys）。
- [ ] API Key 不明文落库（至少服务端加密存储）。
- [ ] 全站 HTTPS（强制跳转 + 安全响应头）。
- [ ] 频率限制：IP + device_id 双维限流（含异常封禁策略）。
- [ ] 输入校验加固：symbol、market、batch 数量、payload 长度、JSON 字段白名单。
- [ ] CORS 白名单收敛为正式域名，不允许 `*`。

## P0-4 可靠性与运维（必须）
- [ ] 进程托管与自恢复（systemd/supervisor/docker restart policy）。
- [ ] 结构化日志（请求ID、设备ID、订阅ID、事件ID、耗时、错误码）。
- [ ] 监控告警：5xx率、响应时延、Webhook失败率、LLM调用失败率、AKShare异常率。
- [ ] 数据备份：SQLite 定时快照 + 保留策略 + 恢复演练。
- [ ] 发布回滚：保留上个稳定版本，一键回退可验证。

## 3. 上线前强烈建议（P1）

- [ ] **契约测试**：`/api/analyze`、`/api/analyze/batch`、`/api/usage`、`/api/subscription`、`/api/webhook/kofi` 全覆盖。
- [ ] **压测**：单机并发峰值、批量分析场景、LLM限流场景。
- [ ] **成本护栏**：每日/每用户 LLM token 上限与熔断策略。
- [ ] **缓存策略**：同标的短时间重复请求缓存，降低 API 成本。
- [ ] **业务看板**：日活、转化率、续费率、请求成功率、平均成本。
- [ ] **客服兜底 SOP**：支付成功未到账、退款、误封、次数异常处理流程。

## 6. 上线前最终 Go/No-Go 清单

仅当以下全部为“是”才允许上线：
- [ ] Ko-fi 验签/幂等/状态机/对账 全部通过联调；
- [ ] 微信真实登录全链路通过真机测试；
- [ ] HTTPS、限流、密钥管理、CORS 白名单完成；
- [ ] 监控告警与备份恢复演练通过；
- [ ] 回滚演练通过；
- [ ] 回归测试与冒烟测试通过；
- [ ] 已确认应急联系人与事故处理流程。

---

## 7. 建议执行顺序（最短路径）

1. 先完成 **Ko-fi P0-1**（最核心收入链路）  
2. 再完成 **微信 P0-2**（用户身份与权益绑定）  
3. 并行完成 **安全/运维 P0-3/P0-4**  
4. 最后做全链路联调 + 回归 + 上线演练

---

## 8. 结论

当前版本可用于内部验证与灰度试用，但还不满足“正式商业上线”标准。  
达到正式可上线的最小闭环是：**Ko-fi 生产化 + 微信正式登录 + 安全运维基线** 全部完成并通过验收。
