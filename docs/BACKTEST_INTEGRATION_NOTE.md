# 回测系统集成说明

## 当前状态

回测脚本 `src/backtest/llm_decision_backtest.py` 已实现LLM决策回测，但与TqSDK的集成有架构性问题。

## 问题描述

### 现有架构
脚本使用**双系统并行**：
1. **脚本自维护账户系统**
   - `self.pos` - 持仓管理
   - `self.balance` - 资金管理
   - `self.trades` - 交易记录
   - ✅ **正常工作**，有准确的交易记录和盈亏统计

2. **TqSDK模拟账户系统**
   - `api.insert_order()` - 下单
   - `api.get_account()` - 获取账户
   - `api.get_position()` - 获取持仓
   - ❌ **未正确集成**，账户显示全为0

### 根本原因
- TqSDK回测需要完整的事件循环集成：
  ```python
  api.insert_order(...)  # 提交订单
  api.wait_update()      # 等待成交
  position = api.get_position()  # 获取持仓
  account = api.get_account()    # 获取账户
  ```
- 当前脚本仅调用了`insert_order()`，但：
  1. 没有在订单后调用`wait_update()`等待成交
  2. 没有使用TqSDK的持仓和账户数据
  3. 自己维护了一套独立的账户系统

### 订单失败原因
1. **"不支持的合约类型"** - 使用了`KQ.m@CZCE.SA`连续合约（TqSim不支持交易）
   - ✅ 已修复：改用`CZCE.SA501`等实际合约

2. **"不在可交易时间段内"** - 下单时间点在非交易时段
   - 原因：4小时周期导致决策时间点（08:00、12:00、20:00）都不在日盘交易时间
   - 影响：即使在交易时段，订单也不会真正成交，因为没有`wait_update()`

## 解决方案

### 方案A：保持现状（推荐）
**优点**：
- 脚本的交易统计完全正常
- 可以快速回测（不受TqSDK回测速度限制）
- LLM决策逻辑已验证可行

**说明**：
- TqSDK仅用于获取历史K线数据
- 交易执行、持仓管理、盈亏计算由脚本自己处理
- 最终统计结果以脚本输出为准（而非TqSDK账户汇总）

**使用方法**：
```bash
python src\backtest\llm_decision_backtest.py --mode llm_direct --symbol CZCE.SA0 --period 240 --count 100
```

查看脚本日志中的交易记录：
```
2025-10-28 22:22:59.745 | INFO  | __main__:_open:587 - 开仓 多 1 手 @ 1241.00（手续费 3.00）
2025-10-28 22:23:05.076 | INFO  | __main__:_close_by_action:679 - 平仓 多 1 手 @ 1246.00，盈亏 +5.00（原因：action）
```

### 方案B：完整TqSDK集成
**需要重构**：
1. 移除脚本自维护的账户系统
2. 完全依赖TqSDK的`get_account()`、`get_position()`
3. 每次决策后调用`wait_update()`等待订单成交
4. 从TqSDK账户读取持仓和盈亏

**缺点**：
- 需要大量代码改动
- 回测速度受TqSDK限制
- 调试复杂度增加

### 方案C：移除TqSDK下单（最简单）
直接移除`_tq_place`相关代码，仅保留数据获取功能。

## 当前推荐

**采用方案A**：
1. 保持现有架构
2. 忽略TqSDK账户汇总（显示全0是正常的）
3. 以脚本自己的交易统计为准
4. 在脚本开始时添加说明：

